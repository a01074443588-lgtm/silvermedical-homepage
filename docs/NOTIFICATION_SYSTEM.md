# 상담 알림 시스템 구조

## 목적

홈페이지 상담 접수는 먼저 데이터베이스에 저장하고, 웹 푸시와 카카오톡 발송은 별도 작업자가 처리합니다. 알림 서비스에 장애가 생겨도 보호자의 상담 접수 자체는 정상 완료됩니다.

## 처리 흐름

1. `/consult/`에서 입력값과 개인정보 동의를 검사합니다.
2. 상담을 저장하고 `SM-날짜-고유문자` 형식의 접수번호를 생성합니다.
3. 저장 완료 후 수신 대상 직원별 알림 작업을 만듭니다.
4. `silvermedical-notification-worker`가 대기 작업을 조회해 발송합니다.
5. 성공·실패·만료 결과를 발송 기록에 남깁니다.
6. 실패 작업은 1분, 5분, 15분, 60분, 180분 간격으로 다시 시도합니다.
7. 상담을 처음 열거나 `접수 확인`을 누르면 이후 재알림을 취소합니다.

## 선택한 방식

- **웹 푸시:** 표준 Web Push와 VAPID 키를 사용합니다. Firebase 계정을 추가하지 않고도 PC Chrome·Edge, 안드로이드 Chrome, iOS 16.4 이상 홈 화면 웹앱을 지원할 수 있습니다.
- **작업 대기열:** 현재 서버에 Redis가 없으므로 SQLite 기반 알림 작업 테이블을 사용합니다. 별도 Docker 작업자가 항상 실행되며 서버 재부팅 후에도 자동 시작합니다.
- **카카오톡:** 카카오 디벨로퍼스 공식 OAuth와 `나에게 보내기` API만 사용합니다. 직원마다 본인 카카오 계정을 따로 연결합니다.

## 추가된 데이터

- `ConsultationAssignment`: 상담 담당자 지정 이력
- `ConsultationStatusHistory`: 열람·확인·담당자·메모·상태 변경 이력
- `StaffNotificationProfile`: 직원별 역할, 담당 급여종별, 수신 상담 유형과 채널 설정
- `PushSubscription`: 직원과 분리된 기기별 푸시 구독
- `KakaoConnection`: 직원별 암호화 카카오 토큰과 연결 상태
- `NotificationConfiguration`: 재알림과 재시도 운영값
- `NotificationJob`: 발송 예정·처리·재시도·실패 상태
- `NotificationDeliveryLog`: 기기별 실제 발송 결과

## 개인정보 보호

알림에는 신청자 마스킹 이름, 연락처 끝 4자리, 접수 시각, 상담 상세 링크만 포함합니다. 전체 전화번호와 상담 본문은 알림 작업·발송 로그에 넣지 않습니다. 상세 내용은 로그인 권한을 확인한 뒤 직원 관리 화면에서만 표시합니다.

카카오 액세스 토큰과 리프레시 토큰은 서버의 암호화 키로 암호화해 데이터베이스에 저장합니다. VAPID 개인키, 암호화 키, 카카오 앱 키는 Git 저장소에 넣지 않습니다.

## 중복 방지와 장애 복구

- 상담·직원·채널·알림 순번을 조합한 고유 키로 같은 작업의 중복 생성을 막습니다.
- 웹 푸시는 기기별 성공 기록을 남기므로 일부 기기만 실패하면 성공한 기기는 건너뛰고 실패한 기기만 재시도합니다.
- 5분 이상 처리 중으로 남은 작업은 중단된 작업자로 판단해 자동 복구합니다.
- 만료된 웹 푸시 주소는 자동 비활성화합니다.
- 직원 로그인 계정을 비활성화하면 해당 직원의 알림, 푸시 기기, 카카오 연결도 함께 중단합니다.

카카오 API는 외부 발송 직후 서버가 갑자기 종료되는 극히 짧은 구간에 동일 메시지가 다시 발송될 가능성을 완전히 제거할 수 없습니다. 상담 접수 유실을 막기 위해 중복 가능성보다 재시도 가능성을 우선한 구조입니다.
